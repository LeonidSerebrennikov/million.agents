1) Объяснение структуры
Структура классическая, есть одна таблица users, на которую уже ссылаются остальные таблицы, в которых задействован пользователь. Это позволит избежать дублирования записей (первичный ключ уникальный и не позволит создать дубликат), а также настроить поведение связанных записей при удалении одной из них (например, при удалении пользователя он удаляется из всех проектов и снимается со всех задач, где он фигурировал, но если он владелец проекта, удаление будет запрещено). 
Во всех таблицах есть поле created_at, которое сохраняет время создания записи.
Есть many-to-many таблица project_members, которая связывает пользователей и проекты (у проекта много пользователей и пользователь может состоять в нескольких проектах).
Также есть ENUMы для статусов и приоритета, задающие жесткий выбор на уровне бд.
Для хранения истории изменений статусов есть отдельная таблица task_log, куда будут записываться логи при изменении статуса задачи. 

2) Индексы
idx_tasks_project_id: большинство запросов так или иначе будут фильтровать задачи по проекту, поэтому индекс на поле project_id необходим
idx_tasks_executor_id: ускорение вывода списка задач, назначенных на пользователя
idx_tasks_status и idx_tasks_priority: фильтрация по статусу и приоритету задач
idx_log_task_id: вывод истории изменений конкретной задачи
idx_task_log_changed_at: сортировка логов по хронологии
idx_project_members_user_id: быстрая проверка состоит ли пользователь в проекте
idx_projects_owner_id: поиск проектов по владельцу
